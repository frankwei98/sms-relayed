# P2 API and Frontend Design

Date: 2026-07-08

## Context

P1 has moved `sms-relayed` to an OpenWrt-first CLI and service model:

- Typed TOML config at `/etc/sms-relayed/config.toml`.
- Non-interactive `sms-relayed run` for init systems.
- Interactive `sms-relayed setup` and `sms-relayed send`.
- Configurable ModemManager `modem_path`.
- Multiple named forwarding profiles.

P2 builds the new API and frontend on top of that foundation. The old `src/web.rs` is legacy code: it exposes an unauthenticated `GET /api` sender and an inline HTML page. P2 replaces it rather than extending it.

## Goals

- Add a password-protected web console.
- Store received SMS history locally.
- Store outbound SMS attempts sent from the CLI or web UI.
- Let the user review received SMS in the browser.
- Let the user reply to a received SMS or send a new SMS from the browser.
- Keep the runtime suitable for OpenWrt and small Linux gateway devices.
- Keep deployment simple: one `sms-relayed` binary and local config/state files.
- Preserve P1 push forwarding behavior.

## Non-Goals

- No multi-user account system.
- No OAuth, SSO, or external identity provider.
- No public-internet hardening beyond a password-protected local UI.
- No built-in TLS certificate management.
- No cloud sync.
- No MMS support.
- No contact-book sync.
- No message deletion in the first P2 slice.
- No separate Node.js runtime in production.

## Recommended Approach

Use a single-process Rust web service with embedded static frontend assets and SQLite storage.

This is the best fit for OpenWrt because the service remains one binary, the frontend does not require a separate runtime, and SQLite gives durable local history without an external database. The API and frontend run in the same `sms-relayed run` process alongside the SMS forwarding loop when enabled in config.

## Considered Approaches

### Approach 1: Single Rust Process with Embedded Frontend

Rust serves JSON API routes and static frontend files. SQLite stores message history. The frontend is built with a small Vite app during release builds and embedded into the binary.

Pros:

- Best OpenWrt deployment story.
- No runtime Node.js dependency.
- Same process can share ModemManager send logic.
- Easy to install with the P1 installer.

Cons:

- Frontend build integration must be added to release workflow.
- Binary or release package grows.

### Approach 2: Rust API plus Separate Frontend Server

Rust only exposes API routes. A separate web server serves the frontend.

Pros:

- Clear separation.
- Frontend iteration is simpler during development.

Cons:

- More moving pieces on OpenWrt.
- Installer and support burden increase.
- Not aligned with the one-binary goal.

### Approach 3: API Only, No Frontend Build System

Keep a hand-written static HTML page embedded in Rust.

Pros:

- Smallest implementation.
- No frontend build tooling.

Cons:

- Hard to make a comfortable SMS history/reply experience.
- Quickly becomes hard to maintain.

Recommendation: Approach 1.

## Runtime Model

P2 extends `sms-relayed run` so it can start two async tasks:

1. SMS forwarding monitor.
2. Web API/frontend server.

The web server starts only when `[api].enabled = true`. Forwarding remains controlled by `[forward].enabled`. This allows these modes:

- Forwarding only.
- API/frontend only.
- Forwarding plus API/frontend.

If both forwarding and API are disabled, `config check` fails with a clear message.

The CLI gains explicit API-aware commands:

```text
sms-relayed run
sms-relayed api serve
sms-relayed config check
sms-relayed config show
```

`sms-relayed run` remains the service command. `sms-relayed api serve` starts only the web service from config and is intended for development and operator debugging.

## Config Model

P2 extends `AppConfig` with an `[api]` section and a `[storage]` section.

```toml
[api]
enabled = true
bind = "0.0.0.0:8080"
password_hash = "$argon2id$v=19$..."
session_secret = "base64-or-hex-random-secret"
cookie_secure = false

[storage]
enabled = true
database_path = "/var/lib/sms-relayed/sms-relayed.sqlite3"
retention_days = 0
```

Rules:

- `api.enabled` defaults to `false`.
- `api.bind` defaults to `0.0.0.0:8080` when API is enabled.
- `api.password_hash` is required when API is enabled.
- `api.session_secret` is generated by the setup wizard when API is enabled.
- `api.cookie_secure` defaults to `false` because OpenWrt/LAN deployments often run plain HTTP behind a trusted network or reverse proxy. Users can set it to `true` when serving behind HTTPS.
- `storage.enabled` defaults to `false`.
- Enabling the API sets `storage.enabled = true` because the web console needs message history.
- `storage.database_path` defaults to `/var/lib/sms-relayed/sms-relayed.sqlite3`.
- `storage.retention_days = 0` means keep history forever.

The setup wizard supports enabling the web console and setting or rotating the password. It must never print the raw password after entry.

## Authentication

P2 uses a single shared password, not usernames.

Password handling:

- The setup wizard prompts for a password twice.
- Store only an Argon2id password hash.
- Reject empty passwords.
- `config show` prints only whether a password is configured.

Session handling:

- `POST /api/login` accepts `{ "password": "..." }`.
- On success, issue an HttpOnly, SameSite=Lax cookie.
- Session cookie is signed with `api.session_secret`.
- `POST /api/logout` clears the cookie.
- All API routes except login and health require a valid session.
- Static frontend routes are allowed, but the app shell shows the login screen until authenticated.

Threat model:

- This protects against casual LAN access and accidental exposure.
- This is not a complete public internet security boundary.
- The README must continue recommending VPN, firewall, reverse proxy auth, or Cloudflare Access for internet exposure.

## Storage

Use SQLite through `rusqlite` with database calls isolated behind `tokio::task::spawn_blocking`.

This is the P2 choice because it keeps SQL explicit, has a small mental model, and avoids introducing compile-time database metadata requirements. The repository layer owns the blocking boundary so route handlers stay async.

Schema:

```sql
CREATE TABLE messages (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  direction TEXT NOT NULL CHECK (direction IN ('inbound', 'outbound')),
  status TEXT NOT NULL,
  phone_number TEXT NOT NULL,
  body TEXT NOT NULL,
  timestamp TEXT NOT NULL,
  storage_type INTEGER,
  modem_path TEXT NOT NULL,
  source_sms_path TEXT,
  reply_to_id INTEGER,
  verification_code TEXT,
  verification_source TEXT,
  error TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE INDEX messages_timestamp_idx ON messages(timestamp DESC);
CREATE INDEX messages_phone_timestamp_idx ON messages(phone_number, timestamp DESC);
```

Message directions:

- `inbound`: received from ModemManager.
- `outbound`: created from CLI or API send flow.

Statuses:

- `received`
- `queued`
- `sent`
- `failed`
- `cancelled`

Inbound storage rule:

- Persist inbound SMS before push forwarding.
- If push forwarding fails, keep the inbound message with `status = 'received'`; forwarding errors are logged, not stored as message failure.

Outbound storage rule:

- Create outbound row before attempting to send.
- Mark `sent` after ModemManager `Send` succeeds.
- Mark `failed` with error text if ModemManager send fails.
- Mark `cancelled` only for flows that explicitly cancel before send.

Retention:

- If `storage.retention_days > 0`, delete messages older than that many days at startup and then once daily.

## API Routes

All JSON routes return structured errors:

```json
{
  "error": {
    "code": "invalid_request",
    "message": "phone_number is required"
  }
}
```

Routes:

```text
GET  /api/health
POST /api/login
POST /api/logout
GET  /api/session
GET  /api/messages
GET  /api/messages/:id
POST /api/messages/:id/reply
POST /api/send
```

### `GET /api/health`

No auth required.

Response:

```json
{
  "ok": true,
  "service": "sms-relayed"
}
```

### `POST /api/login`

No auth required.

Request:

```json
{
  "password": "secret"
}
```

Response:

```json
{
  "ok": true
}
```

Wrong password returns `401`.

### `GET /api/session`

Auth required.

Response:

```json
{
  "authenticated": true
}
```

### `GET /api/messages`

Auth required.

Query parameters:

- `limit`: default `50`, max `200`.
- `before`: optional message id cursor.
- `phone_number`: optional exact phone filter.
- `direction`: optional `inbound` or `outbound`.

Response:

```json
{
  "items": [
    {
      "id": 123,
      "direction": "inbound",
      "status": "received",
      "phone_number": "10010",
      "body": "message text",
      "timestamp": "2026-07-08T12:34:56Z",
      "verification_code": "123456",
      "verification_source": "service"
    }
  ],
  "next_before": 122
}
```

### `GET /api/messages/:id`

Auth required. Returns the full message row.

### `POST /api/messages/:id/reply`

Auth required.

Request:

```json
{
  "body": "reply text"
}
```

Behavior:

- Looks up the original message.
- Sends SMS to that message's `phone_number`.
- Creates an outbound row with `reply_to_id`.

### `POST /api/send`

Auth required.

Request:

```json
{
  "phone_number": "10010",
  "body": "1071"
}
```

Behavior:

- Sends a new outbound SMS.
- Stores the outbound result.

Validation:

- `phone_number` must be non-empty after trim.
- `body` must be non-empty after trim.
- `body` max length is 1000 characters in P2. ModemManager/provider-specific segmentation is outside P2.

## Frontend

The frontend is a compact operational UI, not a marketing page.

Views:

- Login view.
- Inbox/history view.
- Message detail drawer.
- Reply composer.
- New message composer.
- Settings summary view showing API bind address, database path, and storage retention.

Behavior:

- First screen after login is the message history.
- Messages are sorted newest first.
- Inbound and outbound messages are visually distinct.
- Failed outbound messages are visible with error text.
- Verification codes, when detected, are easy to copy.
- Reply action pre-fills the original sender number and links the outbound row to the inbound message.
- The UI refreshes history on demand and polls every 15 seconds while the tab is visible.

Frontend implementation:

- Use a small Vite + React app.
- Build output is served by Axum.
- Embed the built assets into the binary with `rust-embed`.
- The release asset remains one executable binary.

## Code Boundaries

Expected modules:

- `src/web/mod.rs`: Axum router assembly and web server startup.
- `src/web/auth.rs`: password verification and session cookie handling.
- `src/web/api.rs`: JSON route handlers.
- `src/web/static_assets.rs`: frontend asset serving.
- `src/store/mod.rs`: SQLite schema, migrations, spawn-blocking boundary, and message repository.
- `src/message.rs`: shared message/domain types.
- `src/runtime.rs`: starts forwarding and web tasks according to config.
- `src/dbus.rs`: keeps ModemManager send/listen primitives; must not know about HTTP.
- `src/config.rs`: typed P2 config sections and validation.

Avoid putting database code directly in route handlers. Route handlers call a small repository/service layer so storage behavior can be tested without HTTP.

## Data Flow

Inbound SMS:

1. `dbus::monitor_dbus` receives a ModemManager SMS signal.
2. It extracts phone number, text, timestamp, storage type, and SMS object path.
3. Runtime stores an inbound message row.
4. Runtime forwards the SMS to enabled push profiles.
5. Web history shows the stored row.

Web reply:

1. Browser calls `POST /api/messages/:id/reply`.
2. API validates session and request body.
3. API loads the original message.
4. API creates an outbound row with `queued`.
5. API calls `dbus::send_sms`.
6. API updates outbound row to `sent` or `failed`.
7. Browser receives the final outbound message.

CLI send:

1. `sms-relayed send` prompts for number and body.
2. Runtime creates an outbound row if storage is configured.
3. Runtime calls `dbus::send_sms`.
4. Runtime updates outbound row to `sent`, `failed`, or `cancelled`.

## Error Handling

- API disabled and `api serve` called: fail with `api.enabled is false`.
- API enabled without password hash: `config check` fails.
- API enabled with invalid bind address: `config check` fails.
- Database open failure: service startup fails.
- Database migration failure: service startup fails.
- Login failure: return `401` with a generic message.
- Unauthenticated API access: return `401`.
- Missing or invalid JSON: return `400`.
- SMS send failure: return `502` with a structured error and store outbound status as `failed`.
- Message not found: return `404`.
- Push forwarding failure after inbound storage: log and continue.

## Setup Wizard Changes

`sms-relayed setup` asks whether to enable the web console.

If enabled:

1. Prompt for bind address, default `0.0.0.0:8080`.
2. Prompt for web password twice.
3. Generate `api.password_hash`.
4. Generate `api.session_secret`.
5. Set `storage.enabled = true`.
6. Prompt for database path, default `/var/lib/sms-relayed/sms-relayed.sqlite3`.
7. Prompt for retention days, default `0`.

If disabled:

- Keep `[api].enabled = false`.
- Leave `storage.enabled` unchanged if an existing config already has it.
- Do not initialize SQLite at service startup when `storage.enabled = false`.

## Installer and Service Changes

Installer creates:

```text
/var/lib/sms-relayed
```

Permissions:

- `/etc/sms-relayed`: `0700`
- `/etc/sms-relayed/config.toml`: `0600`
- `/var/lib/sms-relayed`: `0700`
- SQLite database: `0600`

OpenWrt procd and systemd still run:

```sh
sms-relayed run --config /etc/sms-relayed/config.toml
```

No separate frontend service is introduced.

## Testing Strategy

Rust tests:

- Config validation for API enabled/disabled.
- Password hash verification.
- Session cookie signing and rejection.
- SQLite migration creates expected tables and indexes.
- Message repository inserts inbound and outbound rows.
- Outbound status transitions from `queued` to `sent` or `failed`.
- API route auth rejects unauthenticated requests.
- API route auth accepts valid session cookie.
- Message list pagination.
- Reply endpoint uses original phone number.

Integration tests:

- Start Axum router with temp SQLite database.
- Login, list messages, send reply through a mocked SMS sender.
- Verify failed send returns structured error and persists `failed`.

Frontend checks:

- Login form renders.
- Message list renders inbound and outbound examples.
- Reply form calls the correct endpoint.
- Long SMS body does not break mobile layout.

Manual OpenWrt validation:

- Run setup and enable API.
- Start `/etc/init.d/sms-relayed`.
- Login from LAN browser.
- Receive an SMS and confirm it appears in history.
- Reply from browser and confirm outbound row becomes `sent`.
- Restart service and confirm history persists.

## P2 Acceptance Criteria

- A fresh OpenWrt install can enable the web console from `sms-relayed setup`.
- `sms-relayed run` can run forwarding and web server in one process.
- A browser can log in with the configured password.
- Received SMS messages are persisted and visible after service restart.
- Browser replies send through ModemManager and are stored as outbound messages.
- Direct browser send creates an outbound message row.
- API routes other than login and health reject unauthenticated requests.
- README documents the web console, password model, storage path, and LAN exposure warning.
